# genCodeDesc Protocol — WHAT & WHY

## ======>>>WHAT IS genCodeDesc<<<======

- `genCodeDesc` is **external revision-level metadata** describing which lines in a commit are AI-generated.
- It is **NOT** repository content. It is produced by a `codeAgent` (such as HuayanCoder) after each revision is created.
- One `genCodeDesc` record describes **one concrete revision**.
- The protocol name is `generatedTextDesc` — it covers both source code (`codeLines`) and document text (`docLines`).

### Identity Key

Each record is indexed by a composite key:

- `repoURL` — canonical repository URL or local path
- `repoBranch` — branch name (Git) or branch path (SVN)
- `revisionId` — commit hash (Git) or revision number (SVN)

### What One Record Contains

| Section | Purpose |
|---|---|
| `SUMMARY` | Aggregate line counts: `totalCodeLines`, `fullGeneratedCodeLines`, `partialGeneratedCodeLines` (and Doc equivalents) |
| `DETAIL` | Per-file array, each with `fileName` and `codeLines`/`docLines` arrays carrying line-level `genRatio` and `genMethod` |
| `REPOSITORY` | Repository identity: `vcsType`, `repoURL`, `repoBranch`, `revisionId` |
| `CREDENTIAL` | Authentication tokens — treated as ignored envelope metadata by the analyzer |

### Line-Level Attribution

- `genRatio` — integer 0–100, percentage of AI attribution for a single line
  - `100` = fully AI-generated
  - `0` = fully human-written
  - `30` = 30% attributed to AI
- `genMethod` — how the line was generated (e.g., `codeCompletion`, `vibeCoding`, `Manual`)
- Lines can be addressed by `lineLocation` (single line) or `lineRange` (`{from, to}`)

## ======>>>WHY genCodeDesc EXISTS<<<======

### The Question We Want To Answer

> At the end of a requested period, in the specified branch snapshot, **what percentage of the live code lines whose current version was added or modified in `[startTime, endTime]` is attributable to AI generation?**

### Why External Metadata Is Needed

- The repository itself knows **which lines survive** and **which revision last introduced each line** (via blame).
- But the repository does **NOT** know whether a line was written by a human or generated by AI.
- That AI attribution knowledge lives in `genCodeDesc` records, produced by the `codeAgent` at commit time.
- The analyzer must **join** these two systems:
  - **Repository** → which lines survive, which revision is their origin
  - **genCodeDesc** → for that origin revision, what is the AI ratio of each line

### The Primary Metric Formula

```
AI_Window_Live_Ratio = Sum(line.genRatio / 100 for live lines whose current origin revision is in [startTime, endTime])
                       / Total_Live_Changed_Lines_In_Window
```

## ======>>>PROTOCOL VERSIONS<<<======

### v26.03 — AI Lines Only (Current Baseline)

- **File**: [Protocols/genCodeDescProtoV26.03.json](Protocols/genCodeDescProtoV26.03.json)
- **DETAIL model**: records only lines with AI attribution (`genRatio > 0`)
- **Used by**: Algorithm A (blame-based) and Algorithm B (diff-replay)
- **Blame**: NOT embedded — blame is discovered at analysis time from the live VCS

### v26.04 — Incremental Add/Delete with Embedded Blame (Planned)

- **File**: [Protocols/genCodeDescProtoV26.04.json](Protocols/genCodeDescProtoV26.04.json)
- **DETAIL model**: INCREMENTAL — each commit records only lines **ADDED** or **DELETED** in that commit, not the full snapshot
- **Used by**: Algorithm C (embedded-blame, pure genCodeDesc, zero VCS access at runtime)
- **Blame**: EMBEDDED — real `git blame`/`svn blame` output captured at write time by the codeAgent
- **Key additions over v26.03**:
  - `changeType` field: `add` or `delete`
  - `blame` object per line: `revisionId`, `originalFilePath`, `originalLine`, `timestamp`, optional `author`
  - `REPOSITORY.revisionTimestamp` — mandatory for AlgC processing order
  - `genRatio=0, genMethod=Manual` for human-written lines (explicit, not omitted)
  - Delete entries use `blame.revisionId + originalFilePath + originalLine` as identity key to remove from accumulated set
  - `blame.originalLineRange` for bulk deletion of contiguous blocks from the same origin

### v26.03 vs v26.04 — KEY Differences

| | **v26.03** | **v26.04** |
|---|---|---|
| **DETAIL model** | Snapshot — records only AI-attributed lines (`genRatio > 0`) | Incremental — records both `add` and `delete` entries per commit, including human lines (`genRatio=0, genMethod=Manual`) |
| **Blame** | NOT embedded — must run live `git blame`/`svn blame` at analysis time | EMBEDDED — real VCS blame captured at write time by codeAgent |
| **VCS access at runtime** | Required | Not required |
| **Diff artifacts at runtime** | Not needed (AlgA) or needed (AlgB) | Not needed |
| **`revisionTimestamp`** | Absent | Mandatory — enables processing order for accumulation |
| **Delete tracking** | No — deletions are invisible in the record | Yes — `changeType=delete` with blame identity key to remove lines from accumulated set |
| **Human lines** | Omitted (absence = human) | Explicit (`genRatio=0, genMethod=Manual`) |

**The single biggest benefit of v26.04**: it makes the genCodeDesc file set **self-sufficient**. A consumer can compute the exact same metric by just parsing v26.04 files in timestamp order — no repository checkout, no VCS CLI, no diff patches. That unlocks air-gapped, edge, and large-scale batch deployments where repository access is expensive or impossible.

**The cost**: correctness now depends entirely on the codeAgent writing accurate blame at commit time. With v26.03, the analyzer can independently verify line origins via live VCS blame; with v26.04, that verification is gone.

### Version Compatibility

- v26.04 is a **superset** of v26.03 at the protocol level.
- The broader fields (`changeType`, `blame`) are forward-compatible but not consumed by Algorithm A or B.
- JSONC (comments) is accepted by the runtime loader for documentation convenience; analyzer output remains standard JSON.

## ======>>>SCOPE DEFINITIONS<<<======

The protocol supports multiple scope levels. Scope determines **what counts as a measured line**.

| Scope | Name | What It Counts | Protocol Field |
|---|---|---|---|
| A | Pure Source Code | Non-blank, non-comment lines in source files (`.c`, `.cc`, `.cpp`, `.cxx`, `.go`, `.h`, `.hpp`, `.java`, `.js`, `.py`, `.rs`, `.ts`) | `codeLines` |
| B | Source + Comments | All non-blank lines (including comments) in source files | `codeLines` |
| C | Documentation Text | Non-blank lines in doc files (`.md`, `.rst`, `.txt`) | `docLines` |
| D | All Text | Union of source and documentation files | `codeLines` + `docLines` |
